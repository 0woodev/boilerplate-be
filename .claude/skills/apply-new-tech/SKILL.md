다음 기술을 이 프로젝트에 도입하려고 한다:

**도입 대상:** $ARGUMENTS

---

## 작업 순서

### 1. 현재 프로젝트 구조 파악

아래 파일들을 읽어 현재 구조를 파악한다:

- `PROGRESS.md` — 현재 진행 현황 및 결정 사항
- `terraform/` 구조 훑기 (있으면)
- `app/` 구조 훑기 (있으면)
- `requirements.txt` — 현재 의존성

핵심 확인 포인트:
- 기존에 유사한 기술이 이미 도입되어 있는가?
- 현재 아키텍처(Lambda, DynamoDB, API Gateway)와 충돌하는가?
- 새 기술이 Lambda cold start, 메모리 제한, 패키지 크기 등에 영향을 주는가?

### 2. 도입 방법 탐색

웹 검색 또는 지식 기반으로 다음을 조사한다:

- 공식 문서 확인 (설치 방법, 기본 설정)
- AWS Lambda와의 통합 패턴 (있으면)
- 대안 기술 목록 (최소 2~3개 옵션)
- 실제 사용 사례 및 트레이드오프

### 3. 도입 방법 비교

아래 기준으로 옵션을 비교한다:

| 기준 | 설명 |
|---|---|
| 복잡도 | 도입·유지보수 난이도 |
| 성능 | Lambda 환경에서 동작 특성 |
| 비용 | 추가 AWS 리소스 비용 |
| 확장성 | 기존 구조 변경 없이 확장 가능한가 |
| 커뮤니티 | 레퍼런스·문서 풍부도 |

### 4. 추천 방향 결정

비교 결과를 바탕으로 **가장 적합한 방법 1개를 추천**한다.

추천 이유:
- 이 프로젝트 구조에 맞는 이유
- 장·단점 요약
- 리스크 및 주의사항

### 5. 구체적 실행 계획 제시

추천 방법을 실제로 적용하는 단계별 계획:

```
STEP 1. 패키지 설치 (requirements.txt 업데이트)
STEP 2. 공통 설정 파일 생성 위치
STEP 3. 첫 사용 예시 코드
STEP 4. Terraform 변경이 필요하다면 변경 내용
STEP 5. 검증 방법 (로컬 테스트, make local 등)
```

실제 코드 스니펫을 최대한 포함한다.

### 6. PROGRESS.md 업데이트 제안

도입 결정이 내려지면 `PROGRESS.md`에 추가할 항목을 제안한다:

```markdown
- [ ] **{기술명} 도입** — {한 줄 설명}
```

---

## 출력 형식

1. **현재 구조 요약** — 기술 도입 전 상태
2. **도입 옵션 비교표**
3. **추천 방향 + 이유**
4. **실행 계획 (단계별 코드 포함)**
5. **PROGRESS.md 업데이트 제안**
